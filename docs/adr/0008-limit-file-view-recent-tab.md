# 0008. File view の Recent タブを最新30件に制限し件数バッジを非表示にする

日付: 2026-04-30

ステータス: 採用

## コンテキスト

File view の Recent タブは、プラグインが記録した最近開いたファイル履歴をそのまま参照して表示していた。履歴は重複しない most-recent-first の順序を保つが、明示的な上限を持たないため、開いたノート数が増えるほど Recent タブの表示対象も増え続ける。

今回の変更では、Recent タブを File view に参加する最新30件の一意なノートに限定する必要がある。ノートを開き直した場合は重複させず先頭へ移動し、30件を超えた古いノートは Recent タブに出さない。Recent は bounded recency list でありカテゴリ総数ではないため、Recent タブだけ件数バッジを表示しない。一方で Inbox、All、In Progress の件数バッジ、フィルター、ピン留め順、除外フォルダー、Markdown 判定は変更しない。

## 決定

最新30件の制限は `RecentFileHistory` の不変条件として扱う。`record` は開いたファイルを先頭へ移動したうえで30件に切り詰め、保存済みの古い長い履歴を読み込む `fromStoredValue` も同じ上限へ正規化する。これにより File view 側はこれまで通り履歴順に `noteCache` へ解決するだけで、Recent タブには最大30件だけが表示される。

タブの件数バッジ表示可否は `FileView` のタブ定義に置く。Recent だけ `showsBadge` を false にし、Inbox、All、In Progress は従来通り件数バッジを描画する。表示可否をタブ定義に含めることで、描画処理はタブごとの表示方針に従うだけにする。

## 代替案

- `FileView` の Recent タブ生成時だけ `slice(0, 30)` する案は採用しない。保存される履歴自体は増え続け、Recent タブを軽く保つ意図が履歴モデルに反映されないため。
- `main.ts` のファイルオープン記録時に上限処理を置く案は採用しない。履歴の一意性と順序を `RecentFileHistory` が既に所有しており、同じ不変条件を同じクラスに集約するほうが自然なため。
- Recent タブのバッジだけ CSS で隠す案は採用しない。不要な DOM を作らず、タブ定義上の仕様として Recent には件数バッジがないことを表すため。

## 影響

- Recent タブは、開いた順序の most-recent-first を保ちながら最大30件だけを表示する。
- 既存の保存値に31件以上の履歴がある場合、読み込み時に最新30件へ正規化される。
- Recent タブには件数バッジが表示されない。
- Inbox、All、In Progress の件数バッジ、ピン留め順、フィルター、対象ファイル判定は変わらない。

## 結果

Recent タブは開いたファイル履歴全体に引きずられず、最新作業に必要な範囲だけを表示する。履歴の上限とタブ表示方針がそれぞれの所有クラスに置かれるため、今後の File view 変更でも Recent 固有の意味を追いやすくなる。
