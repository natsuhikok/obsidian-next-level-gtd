# 0006. ノートビューのファイルピンは現在表示中のノートを対象にする

日付: 2026-04-26

ステータス: 採用

## コンテキスト

ノートビューのファイルピン操作は ADR 0003 で導入され、`NoteFilePinToggle` が `MarkdownView` にピンボタンを追加していた。従来の実装では、ボタン生成時にその時点の `file` をクロージャへ閉じ込めていたため、同じ Markdown タブ内で別ノートへ移動すると、見た目は再描画されても実際の切り替え対象が前のノートに残ることがあった。

今回の変更では、同じタブ内でリンク移動やナビゲーションを行った後でも、ノートビューのピン操作が常に現在表示中のノートだけを対象にし、表示状態もそのノートの pinned 状態と一致する必要がある。File view と共有する既存のファイル名ベースの pin identity は変更しない。

## 決定

ノートビューのファイルピン操作は、クリック時に `MarkdownView` の現在ファイルを読む。`NoteFilePinToggle` はビューごとに DOM 要素を再利用しても、トグル対象の `TFile` を保持し続けない。

再描画時も `MarkdownView.file` を唯一の基準として、ラベル、`aria-pressed`、表示クラスを更新する。ピン切り替え完了後は、操作時に切り替えたファイルを保存したうえで、同じビューの現在表示ノートに対して UI を再評価する。

## 代替案

- `file-open` ごとに必ず新しいアクション要素を作り直す案は採用しない。既存の `MarkdownView` ごとの要素再利用で十分であり、問題は DOM 再利用ではなく古い `file` 参照を保持していた点にあるため。
- ノートビュー専用に現在ノートの pin state を別管理する案は採用しない。File view と状態が分岐し、ADR 0003 の共有状態という前提を崩すため。
- `toggleFilePin` 側でアクティブビューの現在ファイルへ強制的に合わせる案は採用しない。どのファイルを切り替えるかは UI 操作時点の責務であり、保存経路に UI の文脈を混ぜると File view など他経路の意味が曖昧になるため。

## 影響

- 同じ Markdown タブ内で別ノートへ移動した後も、ノートビューのファイルピンは現在表示中ノートだけを切り替える。
- ノートビューのピンボタンのラベル、押下状態、見た目は、常にそのタブで現在表示しているノートの pinned 状態に一致する。
- File view とノートビューは引き続き同じ `settings.pinnedFileNames` を共有し、ファイル名ベースの照合規則も維持する。

## 結果

ユーザーは同じタブ内を移動しても、ノートヘッダーのピン操作が常に今見ているノートに対して安全に働くと期待できる。共有されたファイルピン状態は維持しつつ、ノートビュー固有の stale target 問題を解消する。
