# 0007. Recent 履歴を実際のファイルオープンから更新する

日付: 2026-04-29

ステータス: 採用

## コンテキスト

File view の Recent タブは、最近開いたノートを開いた順に表示する。従来はワークスペースの `file-open` イベントを起点に recent 履歴を更新していたが、このイベントは既に開いているタブへフォーカスを移した場合にも発火し得る。そのため、通常のタブ切り替えだけで recent 履歴の更新判定と File view の再描画が起き、Recent タブの維持コストが高くなっていた。

今回の変更では、既に開いているタブのアクティブ化では recent 履歴を更新しない。一方で、ノートを開く操作では従来どおり対象ノートを recent 履歴の先頭に移動する必要がある。履歴は File view に参加する Markdown ノートだけを対象にし、最大三十件に制限する。

## 決定

Recent 履歴の更新は `WorkspaceLeaf.openFile` の完了後に行う。`file-open` イベントは Note file pin toggle などアクティブビューに追従する UI 更新には使い続けるが、recent 履歴の更新には使わない。

`RecentFileHistory` は recent 履歴の順序、重複排除、最大件数の責務を持つ。保存済み履歴の復元時も同じ最大三十件の制約を適用し、古い保存データから三十件を超える履歴が表示されないようにする。

File view は `active-leaf-change` で一覧を再描画しない。Recent 履歴が変わった場合は `onRecentFileHistoryChange` で再描画し、ファイル内容や設定が変わった場合は従来の変更通知で再描画する。

## 代替案

- `file-open` イベント内で現在のファイルと前回のファイルを比較する案は採用しない。既に別タブで開いているファイルをアクティブ化した場合と、同じファイルを新しく開いた場合の区別が曖昧になるため。
- File view 側で Recent タブ選択中だけ `active-leaf-change` 再描画を抑止する案は採用しない。タブ切り替え時の recent 履歴更新自体は残るため、要求された履歴更新頻度の低減にならない。
- 保存値だけを表示時に三十件へ切り詰める案は採用しない。保存データ自体が肥大化したままになり、stored recent history も最大三十件にする要求を満たせないため。

## 影響

- 既に開いている Obsidian タブへ移動しても Recent 履歴は更新されず、Recent 目的の File view 再描画も発生しない。
- ノートを実際に開いた場合、対象ノートは Recent 履歴の先頭へ移動し、重複は作られない。
- Recent 履歴は保存値と表示値の両方で最大三十件になる。
- Recent タブは件数バッジを表示しない。Inbox、All、In Progress の件数バッジは維持する。

## 結果

Recent タブは最近開いた最大三十件の対象ノートだけを保持する。通常のタブ移動では recent 履歴の保存や File view の recent 再描画が起きないため、タブ切り替え時の余計な処理を減らせる。
