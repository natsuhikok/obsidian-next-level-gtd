# 0004. File view でフィルター結果から開いた後にフィルターを解除する

日付: 2026-04-24

ステータス: 採用

## コンテキスト

File view はタブごとのファイル一覧に対して一時的な文字列フィルターを持ち、目的のノートを絞り込める。従来はその絞り込み結果からノートを開いても `filterText` が残り続け、File view を別リーフに残したまま使う場合に一覧が狭い状態のままになっていた。

今回の変更では、File view の一覧からファイルを開く操作を実行した時点で、非空のフィルターを解除して通常の一覧に戻す必要がある。対象は File view の一覧起点のファイルオープンに限定し、フィルターの一致規則やタブの並び、recent 履歴の扱いは変えない。

## 決定

フィルター解除の責務は `FileView` に置く。`FileView` がフィルター状態を所有しているため、一覧項目のクリックから `openFile` を呼ぶ直前に `filterText` を空文字へ戻し、File view を再描画する。

この再描画は、同じリーフでノートを開く場合には直後にビューが切り替わるため副作用を持たず、別リーフで開く場合には残る File view を即座に未フィルター状態へ戻す。ワークスペース全体の `file-open` イベントではなく一覧起点の操作に閉じることで、他ビューや他経路のフィルター状態を誤って消さない。

## 代替案

- ワークスペースの `file-open` イベントで常に File view のフィルターを消す案は採用しない。File view 以外でファイルを開いた操作まで対象になり、今回の要求範囲を超えるため。
- `active-leaf-change` に合わせてフィルターを消す案は採用しない。ファイルを開かずにリーフを切り替えた場合まで影響し、一覧起点のファイルオープンという条件を満たせないため。
- フィルター文字列だけを更新して再描画しない案は採用しない。別リーフで開いた場合に入力欄と一覧表示が直ちに未フィルター状態へ戻らないため。

## 影響

- File view で絞り込み結果からノートを開いた後、別リーフに File view が残っていても入力欄と一覧は未フィルター状態になる。
- ファイルの一致規則、タブ選択、recent 履歴、他ビューのフィルター状態は変わらない。

## 結果

ユーザーは File view を一時的な検索導線として使った後に、手動でフィルターを消す必要がなくなる。フィルター解除は File view 内のオープン操作に限定されるため、他の画面や経路への影響を広げない。
